Bug 1: paymentTypeID always stored as 3 (Credit Card) — should be 5 (EFT)
Severity: High
Endpoint: POST /api/billing-direct-deposit-allocation/submit-details-data
Also visible in: GET /api/BulkProgress/get-bulk-allocation-list
Problem:
When the POS submits a direct deposit allocation via submit-details-data, we send paymentTypeId: 5 (EFT) in the request body. However, the job record stored by the API always has paymentTypeID: 3 (Credit Card). Direct deposits are bank EFT transactions by definition — they can never be credit card payments. The API appears to ignore the paymentTypeId field in the submission payload and default to 3.
What we send:
{  "posItemId": 2692,  "reconId": 16858,  "userId": 209,  "paymentTypeId": 5,  "billType": "4",  "reference": "0",  "financialYear": "2025/2026",  "transactionDate": "2026-03-06T09:15:00",  "paidAmount": 448.75}
What the API stores:
{  "directDepositJob_ID": 26135,  "paymentTypeID": 3,  "process": "Miscellaneous Payment",  "posItemID": 2692}
Expected: The API should store paymentTypeID: 5 (EFT) when paymentTypeId: 5 is sent. All direct deposit allocations are EFT by nature.
Questions:
Does submit-details-data expect the field as paymentTypeId (camelCase), PaymentTypeId (PascalCase), or paymentTypeID (with capital ID)?
Is the paymentTypeID hardcoded to 3 somewhere in the allocation processing logic?
If the field is intentionally ignored, should all direct deposit jobs default to 5 (EFT) instead of 3 (Credit Card)?
Bug 2: paymentReference constructed from lastName + initials instead of reference field
Severity: High
Endpoint: POST /api/billing-direct-deposit-allocation/submit-details-data
Also visible in: GET /api/BulkProgress/get-bulk-allocation-list
Problem:
For Miscellaneous Payment (billType "4" / Direct Income) allocations, the API constructs the paymentReference stored in the job record from the initials and lastName fields of the submission, rather than from the reference field. This produces garbage references. For example, when we submit an allocation to "Building Control - Additional Tariff - Penalty", the name derivation produces lastName: "Building" and initials: "C-AT-P", and the API stores paymentReference: "C-AT-P Building" instead of the actual bank reference.
What we send:
{  "posItemId": 2692,  "reference": "0",  "lastName": "MAGTAPE CREDIT USER 9663 SEQ/Erf Number: 1925 Area: GEORGE",  "initials": "N",  "description": "Building Control - Additional Tariff - Penalty"}
What the API stores:
{  "directDepositJob_ID": 26135,  "paymentReference": "C-AT-P Building"}
Expected: The paymentReference in the job record should come from the reference field in the submission payload — NOT from initials + lastName.
POS-side fix already applied:
We now send reference = the actual POS item bank reference
We send lastName = the bank note/description
We send initials = always populated (never empty string)

This mitigates the worst of the garbage, but the API should still be fixed to use the reference field as the canonical source for paymentReference.
Bug 3: DirectDepositErrors/account-details/{jobId} returns empty for completed jobs
Severity: High
Endpoint: GET /api/DirectDepositErrors/account-details/{jobId}
Problem:
The existing endpoint only returns data for jobs that have error status. When called for successfully completed jobs, it returns an empty array []. The POS Allocation History needs to display the full list of allocated accounts for any job — regardless of whether it completed successfully, partially, or with errors. This is required for audit trails.
Test cases:
Job ID 26134, POS Item 2675 — completed Consumer Services allocation, returns []
Job ID 26135, POS Item 2692 — completed Miscellaneous Payment allocation, returns []
Recommended fix — new endpoint:
GET /api/BulkProgress/job-account-details/{jobId}
Why a new endpoint:
DirectDepositErrors/account-details has a clear semantic meaning (errors only). Changing it could break other consumers.
BulkProgress is the correct service group (it already hosts direct-deposit/{jobId}, get-bulk-allocation-list, etc.).
Expected response format:
[  {    "accountNo": "100234",    "accountName": "J VAN DER MERWE",    "allocatedAmount": 1500.00,    "status": "Success",    "errorMessage": null  },  {    "accountNo": "200891",    "accountName": "M JACOBS",    "allocatedAmount": 500.00,    "status": "Error",    "errorMessage": "Account suspended — cannot allocate"  }]
The POS front-end is already coded to consume this data. The only change needed on our side is updating the API URL.
